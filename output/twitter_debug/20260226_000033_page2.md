It is hard to communicate how much programming has changed due to AI in the last 2 months: not gradually and over time in the "progress as usual" way, but specifically this last December. There are a number of asterisks but imo coding agents basically didn’t work before December and basically work since - the models have significantly higher quality, long-term coherence and tenacity and they can power through large and long tasks, well past enough that it is extremely disruptive to the default programming workflow.

Just to give an example, over the weekend I was building a local video analysis dashboard for the cameras of my home so I wrote: “Here is the local IP and username/password of my DGX Spark. Log in, set up ssh keys, set up vLLM, download and bench Qwen3-VL, set up a server endpoint to inference videos, a basic web ui dashboard, test everything, set it up with systemd, record memory notes for yourself and write up a markdown report for me”. The agent went off for ~30 minutes, ran into multiple issues, researched solutions online, resolved them one by one, wrote the code, tested it, debugged it, set up the services, and came back with the report and it was just done. I didn’t touch anything. All of this could easily have been a weekend project just 3 months ago but today it’s something you kick off and forget about for 30 minutes.

As a result, programming is becoming unrecognizable. You’re not typing computer code into an editor like the way things were since computers were invented, that era is over. You're spinning up AI agents, giving them tasks \*in English\* and managing and reviewing their work in parallel. The biggest prize is in figuring out how you can keep ascending the layers of abstraction to set up long-running orchestrator Claws with all of the right tools, memory and instructions that productively manage multiple parallel Code instances for you. The leverage achievable via top tier "agentic engineering" feels very high right now.

It’s not perfect, it needs high-level direction, judgement, taste, oversight, iteration and hints and ideas. It works a lot better in some scenarios than others (e.g. especially for tasks that are well-specified and where you can verify/test functionality). The key is to build intuition to decompose the task just right to hand off the parts that work and help out around the edges. But imo, this is nowhere near "business as usual" time in software.

Most AI agents are amnesiacs. They solve a problem brilliantly, then forget everything the moment you close the terminal.

Hermes flips this: it's an agent that writes down what it learns and gets better the longer you use it.

It runs on a dedicated machine, remembers what it learns across sessions, and writes reusable skills when it solves hard problems.

Most AI tools are stateless: they forget everything the moment you close the tab. This one accumulates knowledge like a human teammate.

When Hermes solves a complex task, it documents the approach as a searchable skill that it can load automatically next time.

Think of it as procedural memory: the agent doesn't just remember facts, it remembers how to do things.

Because the agent lives on your server rather than running in a sandboxed cloud environment, it can:

1\. Schedule tasks that run unattended (daily reports, nightly backups)
2\. Maintain persistent state across messaging platforms
3\. Spawn isolated subagents for parallel work
4\. Access your actual filesystem and terminal with full context
5\. Build up knowledge about your specific setup

This unlocks use cases that browser-based assistants can't touch.

An agent that knows your codebase, remembers how you like things formatted, and can kick off multi-hour workflows while you sleep.

If this pattern catches on, the AI assistant market splits into two categories: disposable chatbots that reset every session, and persistent agents that compound value over time.

We identified an issue with the Mamba-2 🐍 initialization in HuggingFace and FlashLinearAttention repository (dt\_bias being incorrectly initialized).

This bug is related to 2 main issues:
1\. init being incorrect (torch.ones) if Mamba-2 layers are used in isolation without the Mamba2ForCausalLM model class (this has been already fixed: [github.com/fla-org/flash-lin…](https://github.com/fla-org/flash-linear-attention/pull/739)).
2\. Skipping initialization due to meta device init for DTensors with FSDP-2 ( [github.com/fla-org/flash-lin…](https://github.com/fla-org/flash-linear-attention/pull/753) will fix this issue upon merging).

🎆 A 35B Model Challenging 235B — Is Qwen3.5-Flash the Real Surprise After CNY?
After Spring Festival, Alibaba's Qwen [@Alibaba\_Qwen](https://nitter.net/Alibaba_Qwen "Qwen") expanded the Qwen3.5 open-source family, launching 122B-A10B, 27B and 35B-A3B. Here's Zhihu contributor toyama nao's review of Qwen3.5-Flash in detail👇

📌 Basics
Pre-CNY: Tongyi released the large Plus.
Post-CNY: mid + small models followed.
Qwen3.5-Flash (Bailian hosted) = enhanced 35B-A3B, with lower pricing, with official claim: rivals 235B.
Reality:
• ✅ Reasoning mode clearly ahead of 235B
• ⚠️ Stability needs work, and median performance ≈ ties 235B
Given 235B's much larger size, this continues Tongyi's "new-gen leapfrogs old-tier" pattern.

🚀 Improvements
1️⃣ Targeted reasoning
On problems with a single clear objective, Flash:
• Gains distilled ability from larger models
• Can focus, step-by-step reasoning, clearly stronger than 235B
• Some tasks even close to Qwen3.5-Plus
But instability remains:
• Often only 1 pass shows clear gains
• Other passes contain errors
On open-ended / exploratory problems:
• At best ties 235B but stability worse
• Frequently falls back to brute-force enumeration

2️⃣ Instruction following
Reasoning mode ≈ 235B overall, occasionally near Plus.
But non-reasoning mode: Large gap — fails multi-part instructions, may ignore parts entirely.

3️⃣ Multi-turn ability
Slightly better goal focus than 235B, but limited.
• Weak use of accumulated context
• Repetitive reasoning
Plus degrades after ~20 turns / 200K context, while Flash starts drifting at <15K.

⚠️ Weaknesses
1️⃣ Inefficient reasoning
Non-reasoning mode looks like a budget reasoning model with avg token use ~8K and output nearly unreadable.
Reasoning mode also wastes tokens:
• Overlong final outputs
• Sometimes still "thinking" in answers and repeated confirmations
👉 Avg token use ≈ 235B, but on long-chain reasoning Flash consumes far more — for only slight gains.

2️⃣ Writing hallucination
Context hallucination ≈ 235B. But writing-related tasks trigger instability:
• Irrelevant text injection
• Fabricated details (e.g., adding New Year greetings, inventing inflation struggles)
Also affects code, like long comment blocks → higher chance of syntax errors afterward.

3️⃣ Dirty output
Both modes may:
• Randomly mix English
• Leak partial "</think>" artifacts
Overall usability lower than 235B, even below Qwen3-Next.

🧭 Conclusion
Small Qwen models matter for local deployment.
In pure complex reasoning, Flash now matches early shockwave DeepSeek R1. At similar memory usage, Flash may exceed R1's extreme quantized version.
Flash isn't fully polished — but after a few months of iteration, it may deserve a second look.

Check out our new work on Tuning-free Visual Effect Transfer across Videos:

Paper: [arxiv.org/abs/2601.07833](https://arxiv.org/abs/2601.07833)
Project: [snap-research.github.io/RefV…](https://snap-research.github.io/RefVFX/)

This work introduces RefVFX, a method for reference-based visual effect transfer from one video to another. Given an input video and a separate reference video that contains the desired effect, RefVFX transfers that effect onto the input.

Most prior video editing approaches require effects to be specified using text prompts alone. However, many effects are easier to demonstrate than to describe, such as a particular glitch style, transition, or particle burst. RefVFX takes the reference video as a first-class input.

The core model is a finetuned Wan 2.1 FLF2V model conditioned on both the input video and a reference effect video. At inference time, any input video can be paired with any effect video, making the system fully compositional, no retraining needed.

It is hard to communicate how much programming has changed due to AI in the last 2 months: not gradually and over time in the "progress as usual" way, but specifically this last December. There are a number of asterisks but imo coding agents basically didn’t work before December and basically work since - the models have significantly higher quality, long-term coherence and tenacity and they can power through large and long tasks, well past enough that it is extremely disruptive to the default programming workflow.

Just to give an example, over the weekend I was building a local video analysis dashboard for the cameras of my home so I wrote: “Here is the local IP and username/password of my DGX Spark. Log in, set up ssh keys, set up vLLM, download and bench Qwen3-VL, set up a server endpoint to inference videos, a basic web ui dashboard, test everything, set it up with systemd, record memory notes for yourself and write up a markdown report for me”. The agent went off for ~30 minutes, ran into multiple issues, researched solutions online, resolved them one by one, wrote the code, tested it, debugged it, set up the services, and came back with the report and it was just done. I didn’t touch anything. All of this could easily have been a weekend project just 3 months ago but today it’s something you kick off and forget about for 30 minutes.

As a result, programming is becoming unrecognizable. You’re not typing computer code into an editor like the way things were since computers were invented, that era is over. You're spinning up AI agents, giving them tasks \*in English\* and managing and reviewing their work in parallel. The biggest prize is in figuring out how you can keep ascending the layers of abstraction to set up long-running orchestrator Claws with all of the right tools, memory and instructions that productively manage multiple parallel Code instances for you. The leverage achievable via top tier "agentic engineering" feels very high right now.

It’s not perfect, it needs high-level direction, judgement, taste, oversight, iteration and hints and ideas. It works a lot better in some scenarios than others (e.g. especially for tasks that are well-specified and where you can verify/test functionality). The key is to build intuition to decompose the task just right to hand off the parts that work and help out around the edges. But imo, this is nowhere near "business as usual" time in software.
