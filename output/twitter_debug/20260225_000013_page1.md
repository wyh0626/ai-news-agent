Important survey on agentic memory systems.

Memory is one of the most critical components of AI agents. It enables LLM agents to maintain state across long interactions, supporting long-horizon reasoning and personalization beyond fixed context windows.

But the empirical foundations of these systems remain fragile.

This new survey presents a structured analysis of agentic memory from both architectural and system perspectives. The authors introduce a taxonomy based on four core memory structures and then systematically analyze the pain points limiting current systems.

What did they find? Existing benchmarks are underscaled and often saturated. Evaluation metrics are misaligned with semantic utility. Performance varies significantly across backbone models. And the latency and throughput overhead introduced by memory maintenance is frequently overlooked.

Current agentic memory systems often underperform their theoretical promise because evaluation and architecture are studied in isolation.

As agents take on longer, more complex tasks, memory becomes the bottleneck. This survey clarifies where current systems fall short and outlines directions for more reliable evaluation and scalable memory design.

A 35 billion parameter model just beat a 235 billion parameter model.

That's not supposed to happen.

Qwen3.5-35B-A3B now outperforms its predecessor that had 6x more total parameters, and it does so while using 7x fewer active parameters per token.

The breakthrough isn't efficiency for efficiency's sake.

It's proof that three specific techniques can compress intelligence better than brute-force scaling:

1\. Hybrid attention layers that mix linear attention (fast, scales to long contexts) with standard attention (accurate, catches nuance) in a 3:1 ratio

2\. Ultra-sparse experts where only 3 billion of 35 billion parameters activate per token, but those 3 billion are chosen by a router trained on higher-quality data

3\. Reinforcement learning scaled across millions of simulated agent environments, not just text prediction

The result is a model architecture where intelligence comes from better routing decisions, not bigger weight matrices.

This unlocks four things that weren't practical before:

1\. Running frontier-class reasoning on a single GPU node instead of a cluster

2\. Serving 1 million token contexts in production without exploding costs

3\. Building agents that can handle complex tool use without the latency penalty of dense models

A naive implementation indeed will use a bit more memory. Lets dig into memory efficiency of deep learning optimizers:

Adam (Kingma 2015), RmsProp (Hinton 2015): 2 buffers, 4 bytes each - 8 bytes per param.

SM3 (Anil et al 2019) / Adafactor (Shazeer & Stern 2018): 1 buffer + sublinear storage for second moments - 5 bytes per param. AdaFactor proposes update clipping to remove momentum, but in practice both are used.

Shampoo (Gupta et al 2018): 5 buffers (momentum, L, R, L\_inv, R\_inv), assuming square params - 20 bytes per param!

Distributed Shampoo (Anil et al 2020, Anil
& Dayma 2022):
-notice L and R are symmetric
\- store L+R as triangular factors plus diagonals
\- notice that L\_inv and R\_inv can be quantized well to int8 (sub diagonal need to be in f32)

3 buffers: 12 bytes per param.

Quantized: 10 bytes per param compared to 8 bytes per param.

But notice: Distributed shampoo paper also indicates that one sided shampoo work just as well for model layers.

For those layers: drops further to 6 bytes per param.

So overall savings from 20 bytes per param
to 6 bytes per param.
